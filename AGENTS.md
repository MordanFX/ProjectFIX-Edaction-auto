# Правила работы с проектом

Этот файл обязателен для Codex/AI-агента. Перед изменениями в проекте нужно сверяться с рабочими `.md`-файлами, но делать это без лишнего шума.

## Обязательный порядок перед кодом

Перед любым изменением кода, базы, фронта, ботов или деплоя:

1. Свериться с `PROJECT_STRUCTURE.md`.
2. Свериться с `CURRENT_STATE.md`.
3. Свериться с `TODO_NEXT.md`.
4. Проверить `git status --short`.
5. Только после этого вносить изменения.

Важно: не выводить рабочие `.md`-файлы целиком в терминал. Для конкретной задачи читать только релевантные разделы через `rg`, `Select-String` или короткий фрагмент файла. Полное чтение всех md делать только по прямой просьбе пользователя.

Если задача консультационная и код не меняется, достаточно опираться на эти файлы как на источник текущего контекста.

## Границы архитектуры

- Telegram и Discord — разные продуктовые потоки.
- Telegram-ученики и Discord-ученики не должны смешиваться в UI, API и бизнес-логике.
- Telegram-бот, Discord-бот и веб-панель не вызывают друг друга напрямую.
- Общая точка интеграции между ними — база данных.
- Общие модели и правила предметной области находятся в `src/course_platform`.
- Telegram Bot API вызывается напрямую через async `httpx`, без bot-фреймворков.

## Backend

- Python 3.12+.
- Ввод-вывод — асинхронный.
- FastAPI используется для веб-панели/API.
- SQLAlchemy используется для моделей.
- Изменения схемы базы — только через Alembic migration.
- Идентификаторы Telegram хранить как `BIGINT`.
- Порядок уроков и попыток сдачи защищать ограничениями базы, а не только кодом.

## Frontend

- React/Vite находится в `frontend`.
- Основная маршрутизация разделов панели — `frontend/src/App.tsx`.
- Новые страницы добавлять как отдельные компоненты в `frontend/src/components`.
- UI должен быть понятен куратору без объяснений в чате.

## Секреты и деплой

- Секреты читать только из окружения/`.env`.
- `.env` не коммитить.
- Не трогать старый отдельный проект на сервере: `/opt/projectfix-bot`.
- Текущий проект на сервере: `/opt/projectfix-education`.
- Перед деплоем проверять, какие сервисы будут затронуты.

## Качество

- Новое поведение сопровождать тестами.
- Целевое покрытие — не ниже 70%.
- Перед завершением кодового этапа выполнять:

```powershell
.\.venv\Scripts\python.exe -m ruff check .
.\.venv\Scripts\python.exe -m pytest --cov
```

Для фронта:

```powershell
npm run build
```

## Непринятые решения

Не фиксировать в бизнес-логике без отдельного подтверждения:

- финальный состав ролей и точные права кураторов;
- автоматическое открытие следующего урока после сдачи или после зачёта, если клиент меняет механику;
- окончательные форматы домашнего задания;
- окончательный способ хранения видео;
- правила платного/пробного доступа;
- правила автоматического снятия куратора с проверки работы.

## Текущий принцип разработки

Двигаться этапами. Не реализовывать большой блок целиком, если пользователь просит идти постепенно.

Каждый этап должен иметь понятный результат, который можно открыть и проверить в панели/боте.
