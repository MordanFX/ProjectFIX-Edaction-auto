# Обучающий Telegram-бот + веб-панель — обзор проекта

> Документ для общего понимания проекта (контекст для разработки).
> Детали и реализация — по этапам, в рабочих сессиях.

---

## 1. Что за проект

Образовательная платформа на базе Telegram: ученик проходит курс из записанных
стримов (видео-уроков), после каждого урока выполняет домашнее задание, сдаёт его
через бота, куратор проверяет и даёт фидбэк, ученик движется дальше по курсу.

Две стороны:
- **Telegram-бот** — для учеников (прохождение курса).
- **Веб-панель** — для кураторов/админа (проверка ДЗ, фидбэк, статистика, управление).

Обе части работают с общей базой данных.

---

## 2. Стек

**Общее:**
- Python 3.12 (бот + бэкенд панели — один язык на сервере)
- PostgreSQL (общая база для бота и панели; выдерживает параллельную работу обоих)
- SQLAlchemy (ORM — модели данных описываются один раз, используются и ботом, и панелью)
- pytest + pytest-asyncio + pytest-cov (тесты, покрытие; цель ≥70%)

**Telegram-бот:**
- httpx (async) — прямые вызовы Telegram Bot API, без библиотек-обёрток
  (полный контроль, та же философия, что в предыдущем проекте)

**Веб-панель:**
- Бэкенд: FastAPI (Python, async, авто-документация API, валидация через Pydantic)
- Фронтенд: React (+ простой UI-набор компонентов)
- Авторизация кураторов: JWT-токены
- HTTPS, защита веб-части (валидация входных данных, разграничение прав)

**Видео-хостинг:** решается отдельно (зависит от размера стримов).
Базовый вариант — видео в закрытом Telegram-канале, бот выдаёт по порядку.
Альтернатива при больших файлах — внешний видеохостинг (ссылки).

---

## 3. Функционал — сторона УЧЕНИКА (Telegram-бот)

Действия ученика и алгоритм:

1. **Старт** — /start, регистрация. Бот показывает курс и текущий прогресс.
2. **Получение урока** — бот выдаёт очередной видео-урок (по порядку, не перескочить).
3. **Получение задания** — после урока бот выдаёт текст домашнего задания.
4. **Сдача ДЗ** — ученик загружает выполненную работу в бота (формат сдачи —
   уточняется: файл / текст / скрин).
5. **Ожидание проверки** — работа уходит куратору; ученик видит статус «на проверке».
6. **Получение фидбэка** — куратор проверил → бот присылает ученику фидбэк и вердикт
   (принято / на доработку).
7. **Следующий урок** — открывается по заданному условию (сразу после сдачи ИЛИ
   после «зачёта» куратора — уточняется).
8. **Прогресс по запросу** — ученик пишет боту и видит: пройдено X из N уроков,
   сколько ДЗ осталось, что впереди.

Принципы: ученик движется строго по порядку; всегда видит, где он и сколько осталось;
ощущение прогресса (статистика, «ты на уроке 5 из 12»).

---

## 4. Функционал — сторона КУРАТОРА / АДМИНА (веб-панель)

1. **Вход** — авторизация (логин/пароль, роли: куратор / админ).
2. **Очередь проверки** — список сданных ДЗ: кто, какой урок, когда сдал, статус.
3. **Проверка работы** — открыть сдачу (файл/текст ученика), посмотреть.
4. **Фидбэк и вердикт** — написать комментарий, поставить «принято» или «на доработку»
   → ученик получает это через бота автоматически.
5. **Статистика** — по ученикам (прогресс каждого), по потоку (сколько на каком уроке,
   сколько работ в очереди, средний прогресс).
6. **Управление курсом** (админ) — уроки, задания, порядок, потоки, доступы
   (в т.ч. бесплатный/ознакомительный доступ).

---

## 5. Архитектура (как всё связано)

```
  УЧЕНИК                     ОБЩАЯ БАЗА (PostgreSQL)            КУРАТОР
  Telegram-бот   <------>   ученики, курсы, уроки,    <------>  Веб-панель
  (Python+httpx)            задания, сдачи, статусы,            (FastAPI + React)
                            фидбэк, прогресс
```

Бот и панель НЕ общаются напрямую — они обмениваются через базу.
Ученик сдал ДЗ (бот → запись в базу) → куратор видит в панели → ставит фидбэк
(панель → запись в базу) → бот забирает и шлёт ученику.

Модели данных (ученики, уроки, сдачи и т.д.) общие — пишутся один раз на SQLAlchemy.

## 6. Поэтапность

**Этап 0 — фундамент.**
Структура проекта, конфиг, подключение к PostgreSQL, модели данных (SQLAlchemy),
правила проекта (аналог CLAUDE.md), базовые тесты.

**Этап 1 — бот: ядро учебного потока.**
Регистрация ученика, выдача уроков по порядку, выдача заданий, приём ДЗ,
прогресс по запросу. Видео — пока заглушка/простая выдача.

**Этап 2 — проверка ДЗ (промежуточный вариант — в Telegram).**
Сданные работы приходят куратору структурированными карточками с кнопками
(Принять / На доработку / Фидбэк) прямо в Telegram. Это даёт РАБОТАЮЩИЙ продукт
целиком — можно показать клиенту и запустить первый поток, ещё до веб-панели.

**Этап 3 — веб-панель: бэкенд (FastAPI).**
API поверх той же базы: авторизация кураторов, очередь проверки, проверка/фидбэк,
статистика. Покрытие тестами.

**Этап 4 — веб-панель: фронтенд (React).**
Интерфейс куратора: список работ, проверка, фидбэк, статистика, управление курсом.

**Этап 5 — видео-хостинг.**
Финальное решение по хранению/выдаче стримов (Telegram-канал или внешний хостинг),
интеграция в бота.

**Этап 6 — деплой.**
Бот + API + панель + PostgreSQL на сервер, домен, HTTPS, автозапуск, мониторинг.

Принцип: каждый этап — законченный рабочий кусок. После Этапа 2 уже есть полностью
функциональный продукт (бот + проверка в Telegram); веб-панель наращивается поверх
без переписывания бота.

## 7. Что ещё уточнить у клиента (по ходу)

- Условие открытия следующего урока (сразу после сдачи / после зачёта куратора).
- Формат сдачи ДЗ (файл / текст / скрин / любой).
- Размер/длина стримов → выбор видео-хостинга.
- Сколько кураторов, нужны ли роли/права.
- Нужна ли оплата/доступы внутри (платные потоки, бесплатный ознакомительный).
